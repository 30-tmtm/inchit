# inchit (가칭) 육아 시간관리 앱

## 베타 출시용 기획서 v2 (실행 문서)

본 문서는 `시간표 → 캘린더 자동 확장`을 중심으로, 베타에서 **사용자친화성(쉽게 만들고 쉽게 쓰는 경험)**과 **실효성(누락 감소 체감)**을 검증하기 위한 MVP 범위/정책/KPI/운영 루프를 명문화합니다.

---

## 1) 베타 범위 확정 (MVP Scope)

### 1.1 1차 핵심 타겟 (베타 한정)

- 0~7세 자녀 부모
- 어린이집/유치원 + 병원/학원/루틴 일정이 반복되는 가정
- 목표: “반복 입력”이 많고, 일정 누락이 스트레스로 이어지는 사용자에게 먼저 PMF 신호를 확보

### 1.2 베타에서 검증할 핵심 가설

- 시간표를 먼저 세팅하면, 월간 캘린더의 “반복 일정 확장”을 통해 사용자의 수동 입력 부담이 줄어든다.
- 자동 일정은 사용자가 “왜/근거가 무엇인지” 이해할 수 있어야 신뢰되고, 실제로 일정 확인/수정으로 이어진다.
- 알림은 최소/정확해야 하고, 사용자가 피로감을 느끼지 않는 기본값이 존재한다.

### 1.3 MVP 필수 기능 (베타 포함)

#### (A) 온보딩 / 아이 프로필

- 아이 프로필 1명 등록
  - 필수: 생년월일
  - 선택: 이름(표시용)
- 목표 UX:
  - 첫 실행 3분 내 “이번 주 반복 일정이 눈에 보이는 상태” 도달

#### (B) 주간 시간표 (메인 입력 화면)

- 시간표 CRUD
  - 생성/수정/삭제
  - 드래그 기반(또는 간편 폼 기반) 시간/요일 배치
- 일정 속성(베타 필수 입력 최소화)
  - 제목(필수)
  - 시간(필수, 10분 단위)
  - 요일 반복(선택이 아니라 “반복을 설정하는 구조”로 제공)
  - 장소(필수)
  - 색상(선택, 미선택 시 자동 부여)
  - 메모/준비물/알림 여부(선택)
- 목표:
  - 사용자가 “반복 루틴”을 1회 세팅하고, 캘린더 확장에서 실제 생활 적용을 바로 경험

#### (C) 월간 캘린더 (자동 확장 화면)

- 시간표 기반 월간 자동 반영
  - 동일 시간표 규칙이 월간으로 확장되어 표시
- 사용자 일정 + 자동 일정 통합 표시
- 자동일정 삭제/수정 가능
- 월간캘린더 전체 화면 + 하단 패널 상세 일정 정보
- 충돌 처리 UI 원칙
  - 자동/수동 일정이 겹칠 경우, 우선 노출 + “사용자 확인”이 가능한 선택지 제공

#### (D) 자동 육아 일정(베타 최소 세트)

- 아이 생년월일 기반 템플릿 규칙 적용
- 최소 포함 이벤트 유형(버전 0.1):
  - 예방접종(템플릿 기반, 데이터 출처/갱신 정책 명시)
  - 건강검진(템플릿 기반, 데이터 출처/갱신 정책 명시)
- 원칙:
  - 초기에는 “정확한 의료 일정”을 API로 동기화하기보다, 신뢰 가능한 범위에서 템플릿 기반 제공

#### (E) 홈 대시보드(요약)

- 홈은 "요약 + 진입" 중심으로 최소 구성
- 다음 일정 1건(가장 가까운 일정) 강조
- 오늘 주요 일정 요약(최대 3개)
- 체크리스트 미니카드: `오늘 체크리스트 (K-DST 기준)` + 진행률 + `보기` 버튼
- 이번 주 요약
- (선택) AI 브리핑은 베타 제외(대체 텍스트 요약 제공)

#### (F) 체크리스트 (K-DST 기반 MVP)

- 영유아 발달정보 체크리스트는 `K-DST` 기준 항목으로 구성
- MVP 범위:
  - 월령 고정 1세트(대근육/소근육/인지/언어/사회성/자조 각 영역별 4가지씩 총 최대 24개 체크리스트 항목) 
  - 체크 토글 + 완료일 저장
- 홈에서는 체크 토글을 제공하지 않고 `보기`로 체크리스트 탭 진입만 제공
- 안내 문구(권장):
  - "본 체크리스트는 K-DST 기준 참고용이며, 진단을 대체하지 않습니다." 혹은 발달의 개인차가 있음을 안내함. 

#### (G) 상황별 체크리스트 추천 (규칙 기반, 조건부 MVP 포함)

- 원칙:
  - 생성형 AI가 항목을 새로 만들지 않고, 승인된 DB 템플릿에서만 추천
  - "AI 추천" 대신 "월령/상황 기반 추천"으로 표기
- MVP 포함 조건(여력 있을 때만):
  - 시나리오 3개 이하로 시작: `어린이집 입소 준비`, `외출 준비`, `병원 방문 준비`
  - 월령 구간 단순화(예: 12~~18, 19~~24, 25~36개월)
  - 추천 항목 수 8~12개 제한
  - 추천 결과는 사용자 수정/삭제 후 저장 가능해야 함
- 운영 원칙:
  - 항목별 `출처/버전/검수일` 관리
  - 참고용 가이드 문구 고지(진단/의학적 판단 대체 아님)

### 1.4 정보구조(탭) 확정

- 하단 탭바(5개):
  - `홈`
  - `주간 시간표`
  - `월간 캘린더`
  - `체크리스트`
  - `마이`
- 정책:
  - `주간 시간표`와 `월간 캘린더`는 탭 분리(기존 pill switch 미사용)
  - 편집 작업은 `주간 시간표`, 확인/조회는 `월간 캘린더`에서 수행

### 1.5 베타 제외(후순위)

- AI 브리핑/생성형 자유추천 체크리스트(초기 정확도/신뢰/비용 리스크)
- 고급 통계 분석
- 외부 제휴/광고/커머스 연동
- 멀티 아이(형제) 완전 지원(우선 단일 아이로 단순화)
- 푸시 알림(초기에는 웹/앱 내 알림 UI 중심으로 시작)

### 1.6 사용자 스토리 (베타 기준, 행동 중심)

- (US1) “가입하자마자 이번 주 반복 일정이 보인다.”
- (US2) “내 생활 패턴(어린이집/병원/학원)을 주간 시간표로 10분 내 세팅한다.”
- (US3) “월간 캘린더에서 자동으로 반복이 확장되어 누락이 줄어든다.”
- (US4) “자동 일정이 왜 생겼는지 확인하고, 필요하면 수정한다.”
- (US5) “홈의 체크리스트 미니카드에서 '보기'로 진입해 K-DST 기준 항목을 체크한다.”

### 1.7 베타에서의 성공 정의(정량 + 정성)

- 정량:
  - 가입 후 24시간 내 첫 시간표 완성률
  - 주간 시간표에서 생성된 반복 일정의 평균 개수
  - 자동 일정 열람률/수정률
  - 2주 잔존율
- 정성:
  - “누락 방지 체감” 5점 척도 + 인터뷰 코멘트(월간 캘린더 사용 경험)

## 2) 자동 일정 신뢰성/면책 정책 (Trust Policy)

### 2.1 정책 목표

- 사용자가 “자동 생성 결과를 맹신”하지 않도록, 동시에 “쓸만한 정보”로 신뢰를 확보
- 의료 관련 기능은 법/윤리 리스크가 있으므로 UI/문구로 최소 안전장치 제공

### 2.2 데이터 출처(베타 버전의 기본값)

베타 0.1 자동 일정은 아래 범주 중 하나로 처리:

- (Type T0) 규칙/템플릿 기반(내부 템플릿)
  - “아이 생년월일 + 월령 구간 + 템플릿 규칙”으로 생성
- (Type T1) 공식 데이터 연동(후속 단계)
  - 정식 연동 전까지는 T0를 기본으로 하고, T1 도입 시 변경 이력/갱신 주기를 명확히 제공

### 2.3 UI에 반드시 노출할 문구(권장 텍스트)

- 자동 일정 상세 화면 상단 배너(예시):
  - “이 일정은 아이 생년월일과 월령 기반 템플릿 규칙으로 자동 생성되었습니다. 의료 조언이 아니며, 실제 진료/기관 안내를 우선 확인하세요.”
- 출처/갱신 정보 영역:
  - “템플릿 버전: v0.1 | 마지막 갱신: YYYY-MM-DD(또는 베타 기간 기준)”

### 2.4 템플릿 갱신 주기(베타 운영 기준)

- 베타 기간 동안 기본값 템플릿은 **크게 흔들지 않고**, 버그/피드백으로 인해 필요한 경우에만 버전업한다.
- 권장 갱신 주기:
  - 2주 1회(또는 심각한 오류 수정 시 즉시)로 템플릿 버전/수정 항목을 변경 로그로 제공

### 2.5 변경 이력/롤백 고지

- 템플릿 변경 시 자동 일정 화면 또는 설정의 “자동 일정 정보”에서 아래를 제공한다.
  - 변경 템플릿 버전(이전 → 현재)
  - 변경 요약(예: 범위/월령 구간/일정 시간/유형 분류)
  - 사용자 선택지(예: 숨김 유지 정책, 삭제 휴지통 복구 가능 여부)

### 2.6 생성 근거(Explainability) 요구사항

- 자동 일정 카드/상세에서 최소 2개를 보여줌
  - “자동 생성 근거: 생년월일/월령 기반”
  - “이 일정 유형: 예방접종/검진/발달 이벤트”
- 사용자가 다음 행동을 할 수 있어야 함
  - “수정(시간/장소/메모)”
  - “자동 일정 제외(해당 유형만 숨김)” 또는 “삭제(휴지통 복구 가능)”

### 2.7 충돌 처리 규칙(베타)

- 자동 일정과 사용자 수동 일정이 같은 시간에 겹칠 때:
  - 기본은 “사용자 일정” 우선 노출
  - 자동 일정은 충돌 배지를 표시하고, 상세에서 “수정/숨김” 선택 유도

## 3) KPI 및 이벤트 트래킹 설계 (Analytics Spec)

### 3.1 베타 핵심 KPI 정의

- KPI1: 온보딩 퍼널
  - 가입 → 아이 프로필 입력 → 첫 시간표 생성
- KPI2: 반복 입력 효과
  - 사용자당 주간 시간표 생성(반복 설정 포함) 개수
- KPI3: 자동 일정 효용
  - 자동 일정 열람률
  - 자동 일정 수정률
  - 자동 일정 “열람 후 24시간 내” 관련 수동 수정 발생률
- KPI4: 잔존
  - 2주 잔존율
- KPI5: 누락 방지 체감
  - 홈/설정에서 5점 척도 설문(주 1회, 최대 2회)

### 3.1.1 베타 초기 목표값(가이드)

- KPI1(첫 시간표): 가입 후 24시간 내 첫 시간표 생성률 `>= 40%`
- KPI2(반복 세팅 효율): 사용자당 주간 반복 일정 생성 개수 `>= 6개`(중앙값 기준)
- KPI3(자동 일정 효용):
  - 열람률 `>= 30%`(자동 일정이 1개 이상 생성된 사용자 중)
  - 수정률 `>= 10%`(열람한 사용자 중)
- KPI4(2주 잔존): `>= 15%`
- KPI5(체감): 누락 방지 체감 설문 `평균 4.0/5 이상` 또는 `4~5점 비율 >= 60%`

> 위 목표값은 첫 코호트(초기 사용자) 결과에 따라 조정합니다. “가설 검증 성공/실패” 판단 기준으로 사용합니다.

### 3.2 최소 이벤트 목록(권장 네이밍: snake_case)

- 온보딩
  - `onboarding_started`
  - `child_profile_created`
  - `weekly_schedule_first_created`
- 주간 시간표
  - `weekly_schedule_event_created`
  - `weekly_schedule_event_updated`
  - `weekly_schedule_event_deleted`
  - `weekly_schedule_recurrence_set`
- 월간 캘린더/자동 확장
  - `monthly_calendar_opened`
  - `auto_event_viewed`
  - `auto_event_edited`
  - `auto_event_hidden`(해당 유형 숨김)
  - `date_detail_opened`
- 홈/알림
  - `home_today_viewed`
  - `notification_settings_updated`
- 체크리스트(베타 포함 시)
  - `checklist_viewed`
  - `checklist_completed`
  - `checklist_reco_applied`
  - `checklist_reco_edited`

### 3.3 이벤트 공통 프로퍼티(필수)

- `user_id`
- `child_id`
- `timestamp`
- `event_type`(자동/수동/숨김 등 구분)
- `schedule_source`(weekly_time_table, auto_template 등)

### 3.4 이벤트별 최소 프로퍼티(예시)

- `weekly_schedule_event_created`
  - `title_present`(bool)
  - `duration_minutes`(number)
  - `recurrence_days`(number, 0~7)
  - `category`(institution, health, learning, routine, other)
- `auto_event_viewed`
  - `auto_type`(vaccination, health_check, development)
  - `age_months_at_generated`(number)
  - `template_version`(string)

### 3.5 측정 시 유의사항

- “열람”을 `상세 진입` 기준으로 일치시키기(화면 스크롤만으로 판정하지 않기)
- 자동 일정은 템플릿 변경 시 성과 비교가 흔들릴 수 있으므로 `template_version`을 반드시 남김

## 4) 가족 공유/권한 모델 및 알림 기본값 (Sharing & Alert Rules)

### 4.1 베타 권한 모델(최소 3역할)

- Owner
  - 모든 설정/수정 권한, 멤버 초대/삭제
- Editor
  - 주간 시간표/체크리스트 편집, 알림 설정(범위 내) 수정
- Viewer
  - 열람만(상세 보기/요약 확인)

### 4.2 권한별 허용 범위(권장 표)

- 아이 프로필:
  - Owner만 수정
  - Editor: 읽기
  - Viewer: 읽기
- 주간 시간표:
  - Owner/Editor: 생성/수정/삭제
  - Viewer: 읽기
- 월간 캘린더:
  - 전 역할: 읽기
- 자동 일정:
  - Owner/Editor: 수정/숨김(삭제는 휴지통 정책 적용)
  - Viewer: 읽기 + 설명 확인
- 알림 설정:
  - Owner/Editor: 범위 내 변경
  - Viewer: 기본값 유지(또는 개인 알림만 선택 가능)

### 4.3 알림 기본값(베타)

- 초기 기본 on:
  - 일정 시작 30분 전
  - 당일 아침 요약
- 기본 off 또는 제한 권장:
  - 동일 유형 반복 알림에 대한 빈도 제한(예: 하루 2회 이상 알림 트리거 시 재정의)
- 사용자 설정 최소 항목:
  - 알림 카테고리 체크(기관/병원/학원/루틴/자동 이벤트)
  - 빈도(보수/기본/자유) 중 택1

### 4.4 알림 문구 톤(사용자 친화)

- 짧고 구체적으로:
  - “오늘 08:30 어린이집 등원(장소: ○○) 30분 전입니다.”
- 의료 관련 자동 일정은 면책 문구 요약:
  - “자동 생성(의료 조언 아님). 확인 필요”

### 4.5 공유 편집 충돌 정책(베타)

- 원칙(최소 구현):
  - `last-write-wins`(가장 최근 저장 내용 우선 적용)
- 사용자 피드백:
  - 저장 후 “다른 보호자가 방금 수정했을 수 있어요. 최신 내용으로 갱신합니다.” 토스트(또는 배너) 표시
- 되돌리기(베타 최소):
  - 수정 후 10~15분 이내에는 “이전 버전으로 되돌리기” 버튼 제공(1단계 롤백)
- 변경 이력:
  - 상세 화면 하단에 “누가/언제/무엇을” 요약 로그 3줄 표시(정밀 로그는 후순위)

### 4.6 알림 빈도 제한 규칙(예시)

- 카테고리별 일일 상한:
  - 기관/병원/학원/루틴/자동 이벤트 각각 “하루 최대 2회 알림” 기본값
- 동일 시간 중복 방지:
  - 같은 분 단위로 2개 이상 알림이 겹치면 1개로 묶어서 표시(우선순위 기준)
- 우선순위:
  - 의료 관련 자동 이벤트 > 기관 도착/시작 > 학습/루틴 > 기타

## 5) 베타 운영 루프 (Beta Ops Loop)

### 5.1 기간/대상

- 기간: 2~4주
- 대상: 20~50명 소규모 부모 그룹
- 목표:
  - 기능 사용 장벽 제거와 누락 방지 체감 검증

### 5.2 주차별 운영 계획(예시)

- Week 1 (설치/온보딩 안정화)
  - 목표: 온보딩 3분 내 “이번 주 반복 일정 화면” 도달
  - 수집: 온보딩 이탈 지점, 첫 시간표 생성 실패 원인
- Week 2 (핵심 사용 루프 검증)
  - 목표: 주간 시간표 → 월간 캘린더 확장 확인
  - 수집: 자동 일정 열람/수정 패턴, 충돌 처리 불만
- Week 3 (신뢰성/알림 튜닝)
  - 목표: 자동 일정 신뢰성 문구 이해도, 알림 피로도 파악
  - 수집: 면책 문구/출처 영역 가독성, 알림 빈도 불만
- Week 4 (베타 종료/다음 우선순위 결정)
  - 목표: KPI 달성/미달 원인 정리 + Phase 2 설계 입력 확보
  - 수집: 인터뷰 결론(“이 앱을 계속 쓸 이유”)

### 5.3 주간 리듬(운영 템플릿)

- 월요일: 버그/피드백 분류(치명/중요/개선)
- 수요일: 우선순위 1회 재정렬 + 지표 확인
- 금요일: 사용자에게 변경 사항 요약 공지(간단히)

### 5.4 피드백 수집 방법(권장)

- 정량: KPI 이벤트 대시보드
- 정성:
  - 주 1회 20~30분 인터뷰(또는 설문 + 후속 질문)
  - 인터뷰 질문 예:
    - “시간표를 어떻게 세팅했나요?”
    - “월간 캘린더에서 뭐가 가장 도움이 됐나요/어려웠나요?”
    - “자동 일정이 믿을만했다고 느낀 이유는 무엇인가요?”
    - “알림이 부담스러웠나요? 있다면 어떤 순간에 부담이었나요?”

### 5.5 베타 종료 산출물

- KPI 리포트(달성/미달)
- 사용자 인용문 5~10개(개선 방향 근거)
- 다음 Phase 2 기능 우선순위(근거 포함)

### 5.6 릴리즈/변경 공지 주기(베타 운영)

- 배포 주기(권장):
  - 매주 1회 “기능/버그 요약” 배포(소규모 개선 포함)
  - 치명 버그는 주기와 무관하게 즉시 핫픽스
- 템플릿(자동 일정 규칙) 변경이 있는 경우:
  - 자동 일정 화면 또는 설정의 “자동 일정 정보”에서 변경 이력 고지(템플릿 버전/변경 요약/영향 범위)

### 5.7 버그/피드백 트리아지 룰(간단 버전)

- 분류 기준:
  - `치명`: 일정 생성/캘린더 확장 기능이 동작하지 않음(데이터 손실 또는 핵심 플로우 중단)
  - `중요`: 일부 사용자에서 반복적으로 기능이 실패(하지만 우회 가능)
  - `개선`: UX/가독성/경미한 버그(기능은 동작)
- 처리 원칙:
  - 치명/중요는 배포 다음 스프린트(48~72h) 안에 우선 반영
  - 개선은 지표/인터뷰에서 반복 언급될 때 우선순위 상승

### 5.8 피드백 캡처 UX(사용자 친화)

- 앱 내 “피드백 보내기” 버튼(간단 폼)
  - 어떤 화면에서, 어떤 행동 후, 어떤 문제가 발생했는지 텍스트 입력
  - 가능하면 스크린샷 첨부(선택)
- 자동 일정 신뢰도 설문(짧게)
  - 예: “자동 일정이 도움이 됐나요?” 3점 척도(예/보통/아니오)
- 의료/개인정보 관련 신고는 최소 수집 원칙 적용

## 6) 확장 아이디어 및 출시 순서 (Post-Beta)

### 6.1 확장 원칙
- 확장 우선순위는 `리텐션 기여 → 신뢰 형성 → 수익화` 순으로 결정
- 베타 핵심 루프(시간표/캘린더/체크리스트)가 흔들리면 확장 기능 출시를 보류
- 감성 요소(캐릭터)는 보조 역할로 유지하고, 일정 관리 효용을 침범하지 않도록 제한

### 6.2 후보 기능과 권장 출시 단계

#### Phase A (베타 직후 우선)
- 성장 기록(타임스탬프) v1
  - 체크/루틴 완료를 `도약 스탬프`로 누적 저장
- 일기장 v1
  - 사진 1장 + 텍스트 250자(초간단 입력)

#### Phase B (핵심 루프 안정화 후)
- 체크리스트 추천 고도화(규칙 기반)
  - 월령/상황 기반 추천 시나리오 확대
  - 사용자 수정 로그를 기반으로 템플릿 개선
- 캐릭터 성장 시각화 v1
  - 스탬프 누적에 따른 반응/외형 변화(경량 게임화)

#### Phase C (데이터 축적 후)
- 성장 발달 정보 탭 고도화
  - 키/체중/머리둘레 입력
  - 성장 그래프 시각화
- 성장 발달 기록 보고서
  - 누적 기록 기반 요약 리포트(PDF 등)

#### Phase D (신뢰와 잔존 확보 후)
- 인앱스토어/커머스 연계
  - 체크리스트 맥락 기반 큐레이션
- B2B 제휴 확장
  - 기관/브랜드 제휴, API 연동 기반 확장

### 6.3 출시 게이트(진입 조건)
- Phase A 진입:
  - 온보딩/첫 시간표 생성 퍼널 안정화
- Phase B 진입:
  - 2주 잔존율과 체크리스트 사용률이 목표 범위 달성
- Phase C 진입:
  - 성장 기록 데이터 누적 및 사용자 입력 품질 확보
- Phase D 진입:
  - 사용자 신뢰 지표(이탈/불만률) 안정화 + 코어 기능 만족도 확보

### 6.4 보류/주의 항목
- 생성형 AI 자유추천은 초기 단계에서 보류(정확도/신뢰 리스크)
- 커머스는 조기 노출 시 제품 인식 훼손 가능성이 있어 후순위 유지
- B2B는 B2C PMF 검증 이전 동시 추진 금지(제품 방향 흔들림 방지)
